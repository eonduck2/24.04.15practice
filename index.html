1. Data-Type: number

정수 형태의 데이터 타입,

프로그램 일반의 영역에서 "정수"를 표현하기 위한 확실한 약속.


주관적으로 체감하는 사용법: 자료형 표현, 연산, 데이터 저장 시 자료형 구분

2. Data-Type: character

한 개의 문자를 저장할 수 있는 데이터 타입,

설명 그대로 "한 개"의 문자를 저장하는 자료형이며, 이는 "프로그래밍 활용 영역"에 있어서 복잡할 수도 있는 "문자열"과 "문자"의 경계를 간단하게 나누어주는 역할을 하고있다고 생각합니다.


주관적으로 체감하는 사용법: 자료형 표현, 데이터 저장 시 자료형 구분

3. Data-Type: string

"문자열"을 저장할 수 있는 데이터 타입

간단한 단어조차, 특정한 조건(문자의 길이)에 따라 String으로 처리될 수 있습니다. 개인적인 생각이지만, 프로그래밍 일반의 영역에서의 String은 친숙한 친구와도 같은 느낌이라고 생각합니다.


주관적으로 체감하는 사용법: 자료형 표현, 활용도가 많다고 느껴지는 문자열 저장, 데이터 저장 시 자료형 구분

4. Data-Type: boolean

값을 참(True)과 거짓(False) 중 한 가지만을 가질 수 있는 데이터 타입

제가 가장 처음 프로그램을 접했을 때, 가장 재미를 느꼈던 자료형입니다. 대부분의 논리 구조를 만들 때, 적극적으로 사용하며 실력 조금씩 늘렸던 기억이납니다. 컴퓨터의 입장에서 정직하지않은 자료형은 없겠지만, 저에게는 "자료형 활용" 영역에 있어서, 상당히 정직하다고 생각했던 자료형입니다.


주관적으로 체감하는 사용법: 자료형 표현, 조건식, 논리 구조 설계, 논리 구조 생성, 프로그래밍 실력 향상 경험치, 데이터 저장 시 자료형 구분

5. Data-Type: object

개인적인 자산을 가질 수 있는 데이터 타입

적으면서 생각해보니 프로그램을 공부하면서 사용했던 거의 대부분의 테크닉이 결국에는 객체로 귀결되었다는 느낌이 듭니다. 객체라는 개념에 왜 집중해야하느냐 라고 묻는다면, 프로그램 영역에서는 두말할 필요가 없고, 프로그램에만 국한되지 않고, 현생에서의 "논리적 사고력 증진" 으로 이어진다고 생각합니다. 적어도 저에게는 보상으로 다가온 것처럼, 모든 것이 객체로 보여지는 "눈"을 얻는다는 것은 어쩌면 저주일수도 있지만, 프로그램을 공부하는 입장이라면 명백한 축복이라고 생각합니다.


주관적으로 체감하는 사용법: ∞

6. Data-Type: array

순서를 가지고있는 객체와 같은 데이터 타입


배열은 사용 방식 자체가 매력적으로 다가올 수 있는 자료형입니다. 순서를 매긴 하나의 묶음의 적극적인 활용도를 기대할 수 있기 때문입니다. 개인적인 생각이지만, 본래의 중요도가 일정 부분 "오브젝트"에게 가려져있다고 생각합니다. 오브젝트는 덩치가 너무 큰 자료형이니까요

주관적으로 체감하는 사용법: 자료형 표현, 다중 데이터 활용 및 처리, 데이터 저장 시 자료형 구분

7. Data-Type: function

특정 동작을 수행하기 위한 명령들을 한데 묶어, 이름을 부여한 자료형

a 함수는 중요도를 물론 따져보면 따져볼 수 있겠지만, 그와는 다른 개념 위에 뿌리를 박아둔 느낌입니다. 모든 자료형과 프로그래밍이 으레 그렇듯, "누가 어떻게 쓰느냐"는 상당히 중요한 토픽입니다. 그 중에서도 개인적으로 생각하는 활용도의 정점이라고 생각합니다. 함수가 결국 오브젝트에 귀결된다고 하더라도, 오브젝트와는 다른 활용도를 가지고 있을 수도 있습니다. 지금 시점이라서 할 수도 있는 생각이지만, "함수를 위한 오브젝트"라는 개념도 통용될 수 있지 않을까? 라는 생각도 듭니다. 누구는 함수를 이용해 닭을 잡고, 누구는 소를 잡을 수도 있겠습니다. 이는 명백한 활용도의 영역에 속해있고, 그렇기때문에, 많은 집중이 필요한 자료형이라고 설명할 수도 있겠습니다.
